## 阅读地图（先选路径）

```text
【主路径】局域网部署
  启 Cloud → 装/激活 Edge → 官方扫码打印至 completed
  （不要求配置 livacloud-demo，也不要求理解容器名 integration-demo）

【可选】验证第三方协议
  在主路径已通的前提下，再配 provider + Demo setup，跑通 Demo 一单
  详见 §6 与 [`第三方接入指南.md`](第三方接入指南.md)

【特例】部署变体（附录 A）
  不启 Demo / Demo 另机 / 只部署 Cloud / 外接 PG·Redis / 前后端怎么连 / 公网
```

**职责边界（排障时用）：**

| 组件 | 负责 |
|------|------|
| Cloud | 认证、任务、文件、管理端、向 Edge 下发预览/打印指令 |
| Edge | 出码、预览 UI、用户确认、**经 IPP 打到打印机** |
| 打印机 / 网络 | IPP 可达、墨粉/纸张、本机驱动与默认打印机 |
| 第三方（若启用） | 自己的 entry 页、选文件、HMAC 提交；**不**直连打印机 |

Cloud **不**直接连打印机。`ipp_unreachable` 一类告警首先查 Edge↔打印机，其次才是 Edge 配置/Cloud 任务下发。

## 1. 产品与目标

### 1.1 是什么

FlyPrint = **Cloud 控制面** + **Edge 一体机终端**。

### 1.2 M0 主目标（必达）

在干净局域网中，接手人能：

1. 启动 Cloud 并登录管理端  
2. 安装并激活 Edge，连上 Cloud 与默认 IPP  
3. **官方扫码打印**完成 1 次（任务 `completed`）——对应门禁 **M0-4b**

### 1.3 M0 可选目标

4. 用自带 Demo（或自有第三方）再跑通一单 ——对应 **M0-4c**（协议验收，非局域网部署本身）

### 1.4 明确不在 M0 范围

- 公网 HTTPS / Keycloak 生产加固  
- CI、OTA、节点 RBAC、Users 完整产品页  
- UNCONFIRMED 静默重打  
- 刷码后「旧手机 Entry」与当前终端会话一致性策略（见 §8）  
- 计费、多租户经营平台

## 2. 架构与主流程

### 2.1 官方打印（主路径）

```text
手机扫码
  → Cloud /entry（upload token → terminal_ticket）
  → 选「官方打印」→ /upload 上传 PDF
  → Cloud 向 Edge 下发 preview_file
  → 用户在 Edge 确认参数
  → Cloud 建 print_job → Edge 经 IPP 打印 → 状态回传
```

### 2.2 第三方打印（可选）

```text
手机扫码 → /entry → 选第三方
  → 跳转对方 entry_url（Compose Demo 或真实业务系统）
  → 第三方 HMAC POST print-requests（带 terminal_ticket）
  → 其后与官方相同：Edge 预览确认 → IPP 打印
  → 可选 callback 通知第三方
```

**终端票据（约 5 分钟）：** 绑定「这一单 ↔ 这台机/打印机」；第三方只拿票据，不拿上传凭证。

### 2.3 三条「地址」不要混

| 谁访问谁 | 用什么地址 | 常见误区 |
|----------|------------|----------|
| **手机 / 浏览器 → Cloud** | `http://<Cloud局域网IP>:8012` | 给手机写 `localhost` |
| **Edge → Cloud** | 同上（`cloud.base_url`） | 写成仅本机可达的地址 |
| **Cloud → 第三方回调** | Cloud **出站能解析** 的基址 | 把手机用的局域网 IP 当成容器回调地址（或反过来） |

Compose **同机自带 Demo** 时，回调常用容器名 `http://integration-demo:8080`——这是 **Cloud 容器访问 Demo 容器**，不是手机地址。自有第三方则填对方真实可达的 callback 基址。详见附录 A.2。

仓库：

| 仓 | 角色 |
|----|------|
| `fly-print-cloud` | API、Admin、Nginx、Compose、可选 Demo、文档 |
| `fly-print-edge` | 一体机运行时与安装包源码 |

## 5. 主路径 SOP（局域网：Cloud + Edge + 官方打印）

### 5.1 启动 Cloud

```powershell
cd fly-print-cloud
docker compose up --build -d
docker compose ps
```

说明：当前 Compose **会顺带拉起** `integration-demo` 容器，但 **主路径验收不依赖它**。不配 provider、不打开 `/integration-demo/` 即可。若希望少起该服务，见附录 A.1。

检查：

- [ ] `http://127.0.0.1:8012/health`
- [ ] `http://127.0.0.1:8012/api/v1/health`
- [ ] 管理端可登录 `admin` / `admin123`

**禁止**在需保留数据时执行 `docker compose down -v`。

局域网主路径一般**不必**改 `EXTERNAL_API_URL` / `ALLOWED_ORIGINS`。  
扫码能否被手机打开，靠 **Edge 的 `cloud.base_url`**，不是靠改 Cloud 的 `EXTERNAL_API_URL`。

### 5.2 安装与激活 Edge

1. 安装 `flyprint-edge-setup-1.0.39.exe`  
2. 配置 `cloud.base_url` = 手机与 Edge 都能访问的 Cloud 根：局域网演示常用 `http://<Cloud局域网IP>:8012`；连公网 Cloud 用 `https://域名`。**勿**给手机链路用 localhost；**勿**用 `https://localhost` 再靠出码改写局域网 IP（HTTPS 证书会失败）。WebSocket 随 base_url 自动为 `ws` / `wss`。  
3. 管理端生成激活码 → Edge 激活 → 节点在线  
4. 发现 IPP 打印机并设为默认 → 管理端可见可用  

### 5.3 官方扫码打印（验收 M0-4b）

1. Edge 出示二维码  
2. 手机扫码 → `/entry` → 选 **官方打印**  
3. 上传 PDF → 回 Edge 预览确认 → 打印  
4. 管理端任务状态到 **`completed`**  

**演示纪律：** Edge 刷新二维码后，手机上旧 Entry 页不要继续点；请重新扫码。

## 6. 可选：Compose 同机 Demo（第三方协议验收）

仅当需要验证「第三方 HMAC 接入」或完成门禁 **M0-4c** 时阅读本节。  
正式对接方用同样协议，换自己的 `entry_url` / 密钥即可——不必使用 `livacloud-demo`。细节见 [`第三方接入指南.md`](第三方接入指南.md)。

### 6.1 适用前提

- 主路径 §5 已通（Cloud + Edge + IPP）  
- Demo 与 Cloud **同一 Compose、同一 Docker 网络**（仓库默认形态）

### 6.2 配置 provider

管理端 → **三方接入** → 创建/编辑：

| 字段 | Compose 同机 Demo 常用值 |
|------|--------------------------|
| code | `livacloud-demo` |
| entry_url | `http://<Cloud局域网IP>:8012/integration-demo/entry`（**手机**打开） |
| callback_base_url | `http://integration-demo:8080`（**Cloud 容器**回调 Demo；不是手机地址） |
| allowed_ip_cidrs | `172.16.0.0/12`（覆盖常见 Docker 桥接网段；**勿写成 127.x**） |
| allowed_file_hosts | `integration-demo` |
| allow_private_file_hosts | **开** |

保存后立即复制 **入站 / 出站 HMAC**（只显示一次）。

### 6.3 Demo setup

打开 `http://127.0.0.1:8012/integration-demo/setup`：

- 密码默认 `demo123`（`INTEGRATION_DEMO_ADMIN_PASSWORD`）  
- 入站 = Demo → Cloud（调 `print-requests`）  
- 出站 = Cloud → Demo（callback）  
- 横幅应显示「已配置」

启用第三方集成依赖 Redis（Compose 默认已带）。

### 6.4 Demo 全流程（验收 M0-4c）

1. Edge 出示二维码 → 手机扫码 → 选 Demo  
2. 模拟 SSO → 选 PDF 提交（注意约 5/10 分钟时效）  
3. Edge 预览确认 → 打印  
4. Demo 页到 **`completed`**

## 8. 排障速查

### 8.1 主路径（Cloud / Edge / 打印机）

| 现象 | 优先责任面 | 检查 |
|------|------------|------|
| 管理端起不来 / API 重启 | Cloud | 加密密钥；`docker compose logs api` |
| 手机扫码打不开 | 网络 / Edge 配置 | 同网段；`cloud.base_url`；防火墙 8012 |
| Edge 不在线 | Edge ↔ Cloud | 激活码；`cloud.base_url`；WS 是否被拦 |
| 任务下发了但打不出 / `ipp_*` | **打印机或 Edge↔IPP** | Edge 本机 IPP、默认打印机、线缆/Wi‑Fi；其次 Edge 日志 |
| 票据/操作过期 | 操作时效 | 终端票约 5 分钟；重新扫码 |
| 刷码后旧手机页仍可点官方 | 已知产品行为 | 应重新扫码（策略待另定） |
| 数据被清空 | 运维操作 | 是否误 `docker compose down -v` |

### 8.2 仅 Demo / 第三方

| 现象 | 检查 |
|------|------|
| `callback_base_url` invalid | 管理端版本；Docker 主机名校验；重建 admin |
| Demo 提交 **HTTP 403** | 出口 CIDR 须覆盖 Demo 源 IP（Compose 常见 `172.18.x` → `172.16.0.0/12`） |
| Demo 提交 **401** | setup **入站**密钥与创建时一致 |
| 文件主机拒绝 | `allowed_file_hosts` / `allow_private_file_hosts` |

常用日志：

```powershell
docker compose logs -f api nginx
# 仅排查 Demo 时再加：
docker compose logs -f integration-demo
```

## 附录 A. 部署变体与特例

> 主路径用默认 Compose 全家桶即可。以下仅在「环境不一样」时查阅。

### A.1 不想跑 / 不验收 Compose Demo

- **什么都不用改：** 不创建 `livacloud-demo` provider、不打开 `/integration-demo/`，官方打印照常。  
- 可选少起容器（Nginx 仍可能代理该路径，未起服务时访问会失败，可忽略）：

```powershell
docker compose up --build -d --scale integration-demo=0
```

- 对接**自有**第三方：按 [`第三方接入指南.md`](第三方接入指南.md) 填对方的 `entry_url` / callback / CIDR / 文件主机，**不要**照抄 `integration-demo` 容器名。

### A.2 Demo 与 Cloud 是否同机

| 形态 | `entry_url`（手机） | `callback_base_url`（Cloud→第三方） | `allowed_ip_cidrs` / file hosts |
|------|---------------------|-------------------------------------|----------------------------------|
| **同机 Compose（默认）** | `http://<局域网IP>:8012/integration-demo/entry` | `http://integration-demo:8080` | Docker 网段 CIDR；host=`integration-demo` |
| **Demo 另机 / 自有系统** | 手机能打开的对方 HTTPS/HTTP 入口 | Cloud **出站能访问** 的对方回调基址 | 对方出口 IP/CIDR；对方文件 URL 主机名 |

原则：**手机地址**与 **Cloud 出站回调地址** 经常不是同一个字符串。

### A.3 只部署 Cloud（暂无 Edge）

可验证：管理端登录、健康检查、部分配置与 API。  
**不能**完成：出码、预览确认、真实打印、M0-4b/4c。  
Edge 是现场闭环的一部分，不是「可有可无的客户端」。

### A.4 外接已有 PostgreSQL / Redis

默认 Compose 把 API 指到服务名 `postgres`、`redis`。

| 依赖 | 默认 | 改用外部时 |
|------|------|------------|
| PostgreSQL | 服务 `postgres` | 需改 `docker-compose.yml` 中 `FLY_PRINT_DATABASE_*`（或不用 Compose 跑 API，改用 `api/config.yaml` + `FLY_PRINT_*`）。仅改 `.env` 里 `POSTGRES_HOST` **不够**（compose 里 host 写死为 `postgres`）。 |
| Redis（第三方集成） | 服务 `redis` | 在 `.env` 设置 `INTEGRATION_REDIS_URL`（TLS 用 `rediss://`）。未用第三方时可暂不关心。 |
| MinIO | 服务 `minio` | 改 `MINIO_ENDPOINT` 等；须保证 API 容器网络可达。 |

改完后重建/重启 API，并确认迁移/建表指向的是预期库（避免连错空库）。

### A.5 前后端怎么连接

**Compose 默认（推荐局域网演示）：**

- 浏览器只访问 Nginx `:8012`  
- 静态管理端 + `/api/`、`/auth/`、`/entry` 同域反代到 `api:8080`  
- 管理端构建参数 `REACT_APP_API_BASE_PATH=/api/v1`、`REACT_APP_AUTH_BASE_PATH=/auth`（见 `.env.example`）  
- **不必**给浏览器配单独的 API 主机名  

**本地拆开开发：**

- API：`api/` 下 `go run`，读 `config.yaml` / `FLY_PRINT_*`  
- Admin：`npm start`，按 `admin` 环境变量指到 API  
- 细节见仓库 [`README.md`](../README.md)「独立开发」

**Edge：** 只配 `cloud.base_url` 指向用户可访问的 Cloud 根（经 Nginx 的那一层），支持 `http`/`https`；长连接自动 `ws`/`wss`。不直连 Postgres/Redis。

出码时若 `cloud.base_url` 含 `localhost`/`127.0.0.1`，Edge 会改写成一体机局域网 IP（便于 http 演示）；**HTTPS 正式部署请直接填证书对应域名**，不要依赖该改写。

### A.6 公网 / 生产

正式对接与局域网特例见 [`第三方接入指南.md`](第三方接入指南.md)。教室 Wi‑Fi 演示不要照抄生产加固清单。
